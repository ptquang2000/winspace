# 02.05 — Worker `PositionWindow`: frame-bounds + DPI compensation

**Labels:** `ready-for-agent`
**Depends on:** 02.04

## What to build

Execution of the `PositionWindow` Effect on the Worker thread — the only place the logical
rect from the core becomes real pixels, flush and DPI-correct. The drop-shadow delta and DPI
never enter the core.

- Add a `PositionWindow` arm to the Worker's `execute` visitor.
- Two live reads: `GetWindowRect(h)` and `DwmGetWindowAttribute(h,
  DWMWA_EXTENDED_FRAME_BOUNDS)`. The gap between them is the invisible drop-shadow border.
- Grow the target rect outward by that per-side delta so the **visible** frame lands exactly
  on `PositionWindow.rect`, then `SetWindowPos(..., SWP_NOZORDER | SWP_NOACTIVATE)`.
- Ensure the process is Per-Monitor-V2 DPI aware (manifest or `SetProcessDpiAwarenessContext`
  at startup) so `rcWork` and `SetWindowPos` share one physical-pixel coordinate space.
- `WindowId → HWND` via `reinterpret_cast<HWND>(std::to_underlying(id))`.

## Acceptance criteria

- [ ] `PositionWindow` moves the window so its extended-frame-bounds edges are flush to the
      target rect (no visible gap from the shadow)
- [ ] Positioning is correct on a high-DPI / scaled Display
- [ ] The fill respects `rcWork` (taskbar stays visible)
- [ ] The compensation delta is computed and applied entirely in the Worker; the core carries
      only the logical rect

## Out of scope

- Batched `BeginDeferWindowPos`/`DeferWindowPos` coalescing (03)
- `SWP_ASYNCWINDOWPOS` hardening against hung apps (03)
